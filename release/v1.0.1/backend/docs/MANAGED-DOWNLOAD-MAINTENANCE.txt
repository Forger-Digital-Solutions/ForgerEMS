Status meanings
---------------
- OK -> URL plus remote checksum coverage should participate
  fully in live revalidation
- OK-LIMITED -> URL can be probed live, but checksum
  confirmation still depends on the pinned manifest hash

Revalidation workflow
---------------------
Run:
  .\\Verify-VentoyCore.ps1 -RevalidateManagedDownloads

This checks:
- each enabled managed file URL still resolves
- each remote checksum source still resolves
- drift is reported without changing the manifest

Archive outputs are written under:
- %LOCALAPPDATA%\\ForgerEMS\\.verify\\managed-download-revalidation\\<timestamp>\\
- %LOCALAPPDATA%\\ForgerEMS\\.verify\\managed-download-revalidation\\latest\\

Expected files:
- managed-download-summary.txt
- managed-download-revalidation.csv
- managed-download-revalidation.txt

Failure guidance
----------------
If a managed download breaks:
1. Confirm whether the artifact moved on the official
   project page, release index, or official release feed.
2. Confirm checksum coverage cleanly from an official
   checksum file, asset digest, or a freshly verified
   hash from the live official artifact.
3. Patch the manifest only when the new URL is still an
   official direct artifact with the same legal/operational
   safety profile.
4. If scope must be narrowed temporarily while the item is
   still otherwise acceptable, keep the official page
   shortcut and classify it as disabled-but-eligible.
5. Demote to review-first or manual-only when the flow
   becomes page-only, account-gated, EULA-sensitive,
   checksum-poor, or provenance becomes ambiguous.
6. Do not patch around a broken upstream with third-party
   mirrors, repacks, scraper URLs, or unofficial checksum
   sources just to keep automation alive.

Demote immediately when
----------------------
- the URL no longer resolves cleanly from the official source
- the artifact can no longer be matched confidently to the
  intended versioned payload
- checksum coverage can no longer be confirmed or pinned
  safely from the official artifact/source
- the upstream flow becomes gated, clickthrough-based,
  account-bound, or otherwise ambiguous

Decision guide
--------------
- high-fragility item drifts -> stop automation for that
  item, confirm the official replacement path/checksum,
  then patch or demote before release use
- pinned-only item changes upstream -> recompute the hash
  from the live official artifact, update the pin, and
  demote if the artifact match is no longer confident
- checksum source disappears -> rely only on a safely
  recomputed pinned hash if the official artifact is still
  clear; otherwise demote
- formerly active item pauses for scope reasons -> keep the
  official page shortcut, classify it as disabled-but-
  eligible, and remove active managed claims
- formerly active item becomes questionable -> convert it to
  review-first or manual-only, preserve the official page
  shortcut, and remove automated download claims

Maintenance cadence
-------------------
- monthly: run the revalidation workflow
- before shipping or rebuilding a toolkit USB
- after any manifest URL/checksum edit
- quarterly: manually review high-fragility items first

Retention policy
----------------
- timestamped revalidation snapshots -> keep 12 months
  plus any snapshot tied to a shipped release
- release verification artifacts -> keep for every shipped
  candidate/stable release while that release is retained
- summary reports -> latest\\ is rolling only; timestamped
  summaries follow the same 12-month + shipped-release rule

Pre-release gate
----------------
Before any candidate/stable build intended for distribution:
1. Run .\\Verify-VentoyCore.ps1
2. Run .\\Verify-VentoyCore.ps1 -RevalidateManagedDownloads
3. Review managed-download-summary.txt
4. Ensure maintenance ranks 1-7 have no unresolved drift
   or provenance/checksum ambiguity
