# ForgerEMS Audit - 2026-04-20

## Scope

This audit focused on the maintainable automation and documentation surfaces present in this workspace:

- `VentoyToolkitSetup/Setup_Toolkit.ps1`
- `VentoyToolkitSetup/Update-ForgerEMS.ps1`
- `VentoyToolkitSetup/ForgerEMS.updates.json`
- `VentoyToolkitSetup/*.txt`
- `VentoyToolkitSetup/ReadME.html`

Artifact-only bundles such as `ForgerTools/BlackForgerBox`, `Tools/Portable/*`, `Drivers/*`, and `MediCat.USB` were inventoried as distribution content rather than rewritten source because their upstream source/build pipelines are not present in this workspace.

## Fixes Applied

- Added compatibility entrypoints: `VentoyToolkitSetup/Setup-ForgerEMS.ps1` and `VentoyToolkitSetup/Setup_USB_Toolkit.ps1`.
- Normalized the documented setup command so the shipped filenames and docs no longer disagree.
- Removed the duplicated in-script manifest template from `Setup_Toolkit.ps1`.
- Made the bundled `ForgerEMS.updates.json` the single source of truth for setup shortcut generation and manifest seeding.
- Expanded the manifest so the updater can now refresh all shortcut pages that setup creates, not just a smaller subset.
- Hardened manifest destination handling so `dest` values cannot escape the selected USB/root path.
- Fixed `-WhatIf` behavior so the updater no longer downloads files during preview mode.
- Fixed `-WhatIf` behavior so remote checksum lookups are also skipped during preview mode.
- Fixed setup preview mode so it no longer opens browser pages while previewing.
- Improved preview logging so setup/update logs say `Would ...` instead of claiming work already happened.
- Increased manifest timeout/archive defaults for large ISO maintenance runs.

## Verification Performed

- Ran `Setup-ForgerEMS.ps1` against a disposable root and confirmed folder creation, shortcut generation, docs generation, and manifest seeding.
- Ran `Update-ForgerEMS.ps1 -WhatIf` against a blank disposable root and confirmed it used the bundled manifest fallback.
- Confirmed preview mode left the preview roots empty after execution.
- Confirmed a malicious manifest destination like `..\\escape.url` is rejected with a root-escape error.

## Remaining Gaps

- The workspace is not a git repository, so there is no native change history, branch workflow, or lock-step release process here.
- `ForgerTools/BlackForgerBox` appears to be a packaged desktop artifact with bundled runtimes and dependencies, but its editable source project is not present in this workspace.
- `Tools/Portable/*`, `Drivers/*`, and `MediCat.USB` include many manually curated third-party binaries whose update cadence is still external to the manifest/updater.
- Rescuezilla's official pages clearly indicate checksums exist in GitHub release assets, but an asset-level checksum file was not directly accessible from the current environment, so that manifest entry still has no pinned SHA-256 value.

## Recommended Follow-Up

- Put the maintainable toolkit files under git so future audits can diff intentional changes from bundled asset churn.
- Track `BlackForgerBox` from its real source repository and treat the packaged folder in this workspace as a release artifact.
- Extend the manifest or add a dedicated inventory script for manually bundled portable tools so stale binary folders are easier to spot on each maintenance pass.
